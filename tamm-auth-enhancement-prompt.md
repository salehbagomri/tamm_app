## المطلوب — ما يجب تطويره

### المستوى الأول 🔴 — ضروري جداً

#### ١ — Splash Screen ذكية

**المشكلة الحالية:** التطبيق يفتح مباشرة على شاشة الدخول بدون أي تحقق.

**المطلوب:**
شاشة Splash تظهر عند فتح التطبيق وتقوم بـ:

```
فتح التطبيق
      ↓
عرض شعار تمّ (1.5 ثانية)
      ↓
التحقق من Supabase session
      ↓
┌─────────────────────────────────────┐
│ لا يوجد session  → شاشة تسجيل الدخول│
│ session موجود + profile ناقص → Onboarding│
│ session موجود + profile مكتمل:       │
│   role = manager   → /manager/dashboard│
│   role = technician → /technician/tasks│
│   role = customer  → /customer/home   │
└─────────────────────────────────────┘
```

**التصميم:**
- خلفية `bgPrimary` داكنة
- شعار "تمّ" بخط Harmattan Bold كبير باللون الأبيض
- نقاط تحميل أنيقة أسفل الشعار
- انتقال سلس (fade) للشاشة التالية

**الملف:** `lib/features/splash/screens/splash_screen.dart`

---

#### ٢ — معالجة أخطاء تسجيل الدخول

**المشكلة الحالية:** لو فشل Google Sign-In لأي سبب، التطبيق يتجمد أو يظهر خطأ تقني.

**الحالات التي يجب معالجتها:**

| الحالة | الرسالة للمستخدم | الإجراء |
|--------|-----------------|---------|
| انقطاع الإنترنت | "تحقق من اتصالك بالإنترنت" | زر "حاول مجدداً" |
| إلغاء المستخدم للدخول | لا شيء — عودة هادئة | لا رسالة خطأ |
| خطأ من Google | "حدث خطأ، حاول مجدداً" | زر "حاول مجدداً" |
| خطأ من Supabase | "حدث خطأ في الخادم" | زر "حاول مجدداً" |

**المبادئ:**
- زر Google يُعطَّل فور الضغط ويظهر مؤشر تحميل
- بعد انتهاء المحاولة (نجاح أو فشل) يعود الزر للحالة الطبيعية
- رسائل الخطأ بالعربي، واضحة وبسيطة — لا رسائل تقنية للمستخدم
- تُسجَّل الأخطاء التقنية في الـ console للمطور فقط

---

#### ٣ — تحسين شاشة Onboarding

**المشكلة الحالية:** لا يوجد تحقق كافٍ من المدخلات.

**التحقق المطلوب:**

**حقل الاسم:**
- لا يقبل فراغاً
- لا يقبل أقل من 3 أحرف
- لا يقبل أرقاماً أو رموزاً

**حقل رقم الجوال:**
- يقبل الأرقام فقط
- يبدأ بـ 7 (أرقام يمنية)
- طوله بين 9 و 10 أرقام
- يُضاف +967 تلقائياً للحفظ في قاعدة البيانات
- التحقق من عدم تكرار الرقم (unique في قاعدة البيانات)

**سلوك الشاشة:**
- زر "ابدأ الآن" مُعطَّل حتى تكون كل الحقول صحيحة
- بعد الضغط يُعطَّل الزر ويظهر مؤشر تحميل
- زر الرجوع مُعطَّل تماماً (WillPopScope)
- لو الحفظ فشل — رسالة واضحة وإمكانية إعادة المحاولة

---

#### ٤ — حذف الحساب

**السبب:** متطلب إلزامي من Google Play لأي تطبيق يستخدم Google Sign-In.

**المسار في التطبيق:**
```
حسابي ← الإعدادات ← حذف الحساب
```

**الخطوات الوظيفية:**
```
المستخدم يضغط "حذف الحساب"
          ↓
نافذة تأكيد: "هل أنت متأكد؟ سيتم حذف كل بياناتك نهائياً"
          ↓
زران: "إلغاء" و "نعم، احذف حسابي"
          ↓
حذف البيانات من profiles
          ↓
إلغاء ربط حساب Google من Supabase
          ↓
تسجيل الخروج
          ↓
شاشة تسجيل الدخول مع رسالة: "تمّ حذف حسابك"
```

**في Supabase:**
- حذف من جدول profiles (CASCADE يحذف البقية)
- استدعاء `supabase.auth.admin.deleteUser(userId)`
- أو Supabase Edge Function لحذف المستخدم من auth.users

---

#### ٥ — تسجيل الخروج مع تأكيد

**المطلوب:**
```
المستخدم يضغط "تسجيل الخروج"
          ↓
نافذة تأكيد: "هل تريد تسجيل الخروج؟"
          ↓
"إلغاء" أو "تسجيل الخروج"
          ↓
مسح الـ session + توجيه لشاشة الدخول
          ↓
مسح كل الـ cached data المحلية
```

#### ٧ — مؤشر التحميل على كل الأزرار

**المبدأ العام في كل التطبيق:**
كل زر يقوم بعملية async يجب أن:
1. يُعطَّل فور الضغط
2. يظهر مؤشر تحميل دوّار بدل النص
3. يعود للحالة الطبيعية بعد انتهاء العملية

**Widget مشترك:**
```
TammButton — يقبل:
  - label: نص الزر
  - onPressed: الدالة
  - isLoading: bool
  - type: primary / secondary / danger
```

هذا الـ Widget يُستخدم في كل التطبيق بدون استثناء.

---

#### ٨ — تجديد الـ Token تلقائياً

**المشكلة:** بعد فترة طويلة من عدم الاستخدام، قد ينتهي صلاحية الـ Token ويُطرد المستخدم فجأة.

**الحل:**
- Supabase يجدد الـ Token تلقائياً — لكن يجب التعامل مع حالة انتهاء الصلاحية بشكل صريح
- في حال انتهاء الصلاحية أثناء الاستخدام:
  - محاولة تجديد صامتة أولاً
  - لو فشلت: توجيه لشاشة الدخول مع رسالة "انتهت جلستك، الرجاء الدخول مجدداً"

#### ١١ — تحديث الدور فوري

**المشكلة:** لو المدير رقّى مستخدماً إلى فني، المستخدم لن يرى الواجهة الجديدة إلا بعد إغلاق التطبيق وفتحه.

**الحل عبر Supabase Realtime:**
```
المدير يغيّر role في قاعدة البيانات
            ↓
Supabase Realtime يرسل التغيير فوراً
            ↓
التطبيق يستقبله ويحدّث الـ Provider
            ↓
GoRouter يعيد التوجيه للواجهة الجديدة تلقائياً
```

## قاعدة البيانات — تعديلات مطلوبة

### تعديل الـ Trigger
الـ Trigger الحالي ينشئ profile — يجب التأكد أنه يجلب `avatar_url` من Google metadata:

```sql
-- في الـ trigger:
new.avatar_url = (new.raw_user_meta_data->>'avatar_url');
```

### Supabase Edge Function لحذف المستخدم
```
اسم الـ Function: delete-user
تستقبل: { userId }
تحذف من: auth.users (CASCADE يكمل الباقي)
تُستدعى من: auth_repository.dart
```

### RLS Policy إضافية
```sql
-- المستخدم يستطيع حذف بياناته فقط
create policy "user can delete own profile"
on profiles for delete
using (auth.uid() = id);
```

## معايير الجودة — ثابتة لكل شاشة

كل شاشة في هذا القسم يجب أن تحقق:

**١ — حالات التحميل:**
- كل عملية async عندها loading state
- الـ UI لا يتجمد أبداً

**٢ — حالات الخطأ:**
- كل خطأ محتمل له رسالة عربية واضحة
- دائماً يوجد زر "حاول مجدداً"
- لا رسائل تقنية تظهر للمستخدم

**٣ — حالات الفراغ:**
- لو لم توجد بيانات — رسالة واضحة وأيقونة

**٤ — RTL:**
- كل النصوص والعناصر من اليمين لليسار
- `Directionality(textDirection: TextDirection.rtl)`

**٥ — الخط:**
- Harmattan لكل النصوص بدون استثناء

**٦ — الألوان:**
- من AppColors فقط — لا ألوان مكتوبة مباشرة
